Architecture · Core Principles
Cloud Architecture Design Principles
The foundational thinking patterns that separate great cloud architectures from those that fail at scale, cost too much, or collapse under load.
🏗️ The AWS Well-Architected Framework defines 6 pillars of architectural excellence. Every best practice below maps to at least one pillar.
Principle 01
Design for Failure
Assume every component will fail. Design systems where failure of any single component does not cause full system failure. Use redundancy, retries, and graceful degradation at every layer.
Multi-AZAuto ScalingHealth Checks
Principle 02
Decouple Components
Replace tight dependencies with loose coupling using queues, event buses, and APIs. This prevents cascading failures and allows each component to scale independently.
SQSSNSEventBridgeAPI Gateway
Principle 03
Scale Horizontally
Prefer adding more small instances over upgrading to one large instance. Horizontal scaling is cheaper, more fault-tolerant, and avoids single points of failure.
Auto Scaling GroupsECSLambda
Principle 04
Implement Elasticity
Provision exactly the resources you need at each moment. Scale up for peak demand, scale down when demand drops. Never over-provision statically what can be provisioned dynamically.
Auto ScalingLambdaAurora Serverless
Principle 05
Use Managed Services
Offload undifferentiated heavy lifting to AWS managed services. Don't manage your own message queues, databases, or caches when AWS can do it better, cheaper, and with higher availability.
RDSElastiCacheSQSMSK
Principle 06
Everything as Code
Define all infrastructure as code. Manual console changes are untrackable, non-repeatable, and dangerous. IaC enables version control, peer review, and disaster recovery of your infrastructure.
CloudFormationCDKTerraform
✅ ALWAYS DO
Deploy across minimum 2 Availability Zones
Use load balancers in front of every compute tier
Implement health checks on all services
Use managed services over self-managed where possible
Define and test your RTO and RPO targets
Version-control all infrastructure as code
❌ NEVER DO
Deploy production workloads in a single AZ
Hardcode configuration values in application code
Allow direct SSH access to production servers
Over-provision "just in case" without auto-scaling
Skip testing disaster recovery procedures
Make manual changes to production environments
Architecture · Reliability
High Availability Best Practices
High availability (HA) means the system remains operational and accessible despite component failures. Target: 99.9% (8.7h downtime/yr), 99.99% (52min/yr), or 99.999% (5min/yr) — your architecture determines which is achievable.
1
Multi-AZ Deployments
Deploy all stateful components (databases, caches) in Multi-AZ mode. AWS automatically handles failover. For RDS: enable Multi-AZ (synchronous replication, automatic failover in 60–120 seconds). For Aurora: built-in multi-AZ across 3 AZs by default.
RDS Multi-AZAuroraElastiCache Multi-AZ
2
Elastic Load Balancing
Always place a load balancer in front of compute tiers. ALB for HTTP/HTTPS (Layer 7), NLB for TCP/UDP (Layer 4), GLB for security appliances. Load balancers perform health checks and route traffic away from unhealthy instances automatically.
ALBNLBHealth Checks
3
Auto Scaling Groups
Define minimum, desired, and maximum instance counts. Configure scaling policies based on CPU, memory, custom CloudWatch metrics, or predictive scaling. Always set a minimum of 2 instances across 2 AZs for any production workload.
Auto ScalingCloudWatch AlarmsTarget Tracking
4
Circuit Breaker Pattern
Implement circuit breakers between services to prevent cascading failures. When a downstream service fails, the circuit breaker "opens" and returns a fallback response instead of waiting for timeouts. Use AWS App Mesh or application-level libraries.
App MeshStep FunctionsRetry Logic

Architecture · Performance
Scalability Best Practices
Scalability means your system can handle growth — in users, data, or traffic — without redesign. Build for 10x your current load from day one.
Practice 01
Cache Aggressively
Implement caching at every layer: CloudFront (CDN), ElastiCache (application), DAX (DynamoDB). Caching reduces database load by 80–95% for read-heavy workloads.
CloudFrontElastiCache RedisDAX
Practice 02
Database Read Replicas
Offload read traffic to read replicas. RDS supports up to 5 replicas. Aurora supports up to 15. Direct all SELECT queries to replicas; writes go to primary. This multiplies your effective database throughput.
RDS ReplicasAurora Global
Practice 03
Asynchronous Processing
Move any non-real-time work to queues and process asynchronously. User doesn't wait; system handles load independently. Classic pattern: API → SQS → Lambda/Worker.
SQSLambdaECS Tasks
Practice 04
Content Delivery Network
Serve static assets (images, JS, CSS) from CloudFront edge locations, not your origin servers. CloudFront has 450+ edge locations. This reduces latency by 60–80% and eliminates origin load for static content.
CloudFrontS3 OriginLambda@Edge
Architecture · Networking
VPC & Networking Best Practices
Your network design is your security perimeter and your performance foundation. Bad network design is almost impossible to fix later without significant disruption.
Practice 01
Plan Your CIDR from Day One
Design your IP addressing before creating any VPCs. Use non-overlapping CIDRs if you'll ever peer VPCs or connect to on-premises. Recommended: /16 per VPC, /24 per subnet. Changing later is extremely painful.
VPC CIDR PlanningRFC 1918
Practice 02
Public / Private / Data Subnet Tiers
Three-tier subnet design: Public (load balancers, NAT gateways only), Private (application servers, Lambda), Data (databases, ElastiCache). Nothing that holds sensitive data should ever be in a public subnet.
Public SubnetsPrivate SubnetsNAT Gateway
Practice 03
Use AWS PrivateLink for Service Access
Access AWS services (S3, DynamoDB, SSM, etc.) via VPC Endpoints / PrivateLink rather than through the public internet. This keeps traffic on the AWS backbone, improves security, and reduces NAT gateway data costs.
VPC EndpointsPrivateLinkGateway Endpoints
Practice 04
Transit Gateway for Multi-VPC
When you have more than 3 VPCs, use Transit Gateway instead of VPC peering mesh. TGW simplifies routing, scales to thousands of VPCs, and supports routing policies between VPCs.
Transit GatewayRoute TablesVPC Peering

Practice 01
S3 Lifecycle Policies
Automatically transition objects to cheaper storage classes as they age. Standard → Standard-IA (30 days) → Glacier Instant (90 days) → Glacier Deep Archive (180 days). This alone can reduce S3 costs by 60–80%.
S3 LifecycleIntelligent Tiering

Always enable versioning on critical S3 buckets. Combine with MFA Delete to prevent accidental or malicious deletion of object versions. This is your last line of defense against ransomware in S3.

Architecture · Resilience
Disaster Recovery Best Practices
Disaster recovery (DR) is not optional — it is a business continuity requirement. Define RTO (Recovery Time Objective) and RPO (Recovery Point Objective) first, then choose the appropriate strategy.
🔄 The 4 DR Strategies — Choose Based on RTO/RPO
StrategyRTORPOCostHow It Works
Backup & RestoreHoursHoursLowRegular backups to S3/AWS Backup. Restore from backup during disaster.
Pilot Light10–30 minMinutesMediumCore infrastructure pre-built in DR region, scaled down. Scale up on disaster.
Warm StandbyMinutesSecondsHighScaled-down but functional environment in DR region, continuously running.
Active-Active~0~0HighestFull production in 2+ regions simultaneously. Route53 routes traffic to healthy region.
⚡ Key Rule: Define RTO and RPO with the business first — never with IT first. A business might accept 4-hour RTO for a reporting system but require 15-minute RTO for a payment system. The DR strategy must match the business requirement, not the other way around.
Security · Identity
Identity & Access Management Best Practices
IAM misconfiguration is the #1 cause of cloud security breaches. These practices apply to every AWS account, every workload, without exception.
🚨 The most common cloud breach vector: Overly permissive IAM roles and exposed access keys. These 5 practices eliminate the majority of IAM risk.
Practice 01
Least Privilege Always
Grant only the permissions required for the specific task. Never use wildcards (*) in production policies. Start with read-only, add write permissions only when explicitly required and justified. Review permissions quarterly using IAM Access Analyzer.
IAM Access AnalyzerPermission Boundaries
Practice 02
Never Use Root Account
The root account has unlimited access to everything and cannot be restricted by SCPs or permission policies. Create it once, enable MFA, store credentials in a vault, and never use it again for day-to-day operations. Create separate admin IAM users/roles.
MFA on RootIAM Admin Users
Practice 03
Eliminate Long-Term Access Keys
Access keys (AKID/SECRET) are permanent credentials that can be stolen, leaked in code, or brute-forced. Replace with IAM Roles for EC2, Lambda, ECS. Use AWS SSO / IAM Identity Center for human access. Rotate any remaining keys every 90 days maximum.
IAM RolesIAM Identity CenterSecrets Manager
Practice 04
Enable MFA Everywhere
Enforce MFA for all human IAM users and the root account. Use IAM condition keys (aws:MultiFactorAuthPresent) to require MFA for sensitive actions. Consider hardware MFA (YubiKey) for privileged accounts.
Virtual MFAHardware MFAIAM Conditions
Practice 05
Use IAM Roles for Service-to-Service
When an EC2, Lambda, or ECS task needs to access another AWS service (S3, DynamoDB, etc.), always use an IAM Role — never pass credentials as environment variables. The role is automatically rotated by AWS; credentials in code are not.
Instance ProfilesExecution RolesTask Roles
Practice 06
Service Control Policies (SCPs)
In multi-account environments, use SCPs at the AWS Organizations level to set maximum permission boundaries. SCPs are guardrails — they cannot grant permissions, only restrict them. Use to prevent disabling CloudTrail, creating public S3 buckets, or leaving specific regions.
AWS OrganizationsSCPsPermission Boundaries
✅ IAM DO
Enable AWS CloudTrail in all regions
Use IAM roles, never access keys on EC2/Lambda
Enable MFA for all human users
Regularly review unused permissions with Access Analyzer
Store secrets in Secrets Manager, not environment variables
❌ IAM NEVER
Use AdministratorAccess policy for application roles
Hardcode credentials in source code
Share IAM user credentials between team members
Leave access keys older than 90 days active
Disable CloudTrail to "save costs"
Security · Data
Data Protection Best Practices
Protect data at rest and in transit. Every piece of data in your cloud environment should be encrypted — the question is which key management approach is appropriate for each classification level.
Practice 01
Encrypt Everything at Rest
Enable encryption on all AWS storage services. S3: server-side encryption (SSE-S3, SSE-KMS, or SSE-C). EBS: encrypted volumes. RDS: encrypted at instance creation (cannot be enabled after). ElastiCache: encryption at rest. KMS adds audit trail via CloudTrail.
AWS KMSS3 SSEEBS Encryption
Practice 02
Enforce HTTPS / TLS in Transit
All data in transit must be encrypted. ALB: enforce HTTPS, redirect HTTP. API Gateway: HTTPS only. S3: enforce SecureTransport condition in bucket policy. RDS: require SSL connections. Never allow unencrypted communication for sensitive data.
ACM CertificatesTLS PoliciesS3 Bucket Policy
Practice 03
Block All Public S3 Access
Enable "Block Public Access" at the account level AND at each bucket level. Use AWS Config rule s3-bucket-public-read-prohibited to continuously monitor. S3 data breaches are almost always caused by accidentally public buckets.
S3 Block Public AccessAWS ConfigSecurity Hub
Practice 04
Use AWS Secrets Manager
Never store database passwords, API keys, or certificates in code, environment variables, or SSM Parameter Store unencrypted. Secrets Manager provides automatic rotation, fine-grained access control, and full audit trail for all secret access events.
Secrets ManagerKMSRotation Lambda
Security · Network
Network Security Best Practices
Defense in depth — multiple layers of network controls so that no single misconfiguration opens a critical path to your data.
Practice 01
Security Groups as Firewalls
Security Groups are stateful firewalls at the instance level. Follow the principle of least privilege: open only the ports required from only the sources required. Reference other Security Groups (not IP ranges) for internal traffic. Never open 0.0.0.0/0 on sensitive ports.
Security GroupsSG References
Practice 02
Network ACLs for Subnet Boundaries
Use NACLs as a stateless second layer of defense at subnet boundaries. Block known malicious IP ranges. Explicitly deny traffic that should never cross subnet boundaries. NACLs process rules in order — lower rule number = higher priority.
Network ACLsSubnet Isolation
Practice 03
AWS WAF for Web Applications
Deploy AWS WAF in front of ALB, CloudFront, and API Gateway. Enable AWS Managed Rules (covers OWASP Top 10). Add rate-limiting rules to prevent DDoS. Enable AWS Shield Standard (free) and consider Shield Advanced for critical applications.
AWS WAFShieldManaged Rules
Practice 04
No Direct Internet to Private Resources
Databases, application servers, and caches must never have public IP addresses. Outbound internet access from private subnets must go through a NAT Gateway. Inbound must go through a load balancer. Use AWS Systems Manager Session Manager instead of bastion hosts for admin access.
NAT GatewaySession ManagerPrivate Subnets
Security · Detection
Threat Detection & Response Best Practices
Assume breach. The question is not whether you will be attacked, but whether you will detect it quickly and respond effectively. Automated detection is essential — manual log review at scale is impossible.
1
Enable AWS GuardDuty in All Accounts & Regions
GuardDuty uses ML and threat intelligence to detect: compromised EC2 instances, unusual API calls, cryptocurrency mining, data exfiltration attempts, and more. Cost: ~$1–3/account/month for most workloads. The first 30 days are free. No reason not to enable it.
GuardDutyThreat IntelligenceML Detection
2
Centralize Logs with CloudTrail & CloudWatch
Enable CloudTrail in all regions with a multi-region trail. Send logs to a dedicated S3 bucket with MFA Delete enabled. Enable CloudTrail log file validation (detects if logs are tampered with). Ship logs to CloudWatch Logs for real-time alerting on critical events.
CloudTrailCloudWatch LogsLog Integrity
3
AWS Security Hub as Your Security Dashboard
Security Hub aggregates findings from GuardDuty, Inspector, Macie, IAM Access Analyzer, and Firewall Manager into a single prioritized view. It scores your environment against CIS AWS Foundations Benchmark and PCI DSS automatically.
Security HubCIS BenchmarkAggregated Findings
4
Automate Incident Response
Don't rely on humans to respond to security alerts at 3 AM. Use EventBridge rules to trigger Lambda functions on critical GuardDuty findings. Example: GuardDuty detects compromised EC2 → EventBridge → Lambda → isolates instance (removes from SG) → notifies SNS → creates Jira ticket.
EventBridgeLambdaSNSAutomation

Cost Optimization · Foundation
Cost Visibility Best Practices
You cannot optimize what you cannot see. Cost visibility is the foundation of every other cost practice — without it, you are guessing.
Practice 01
Enable AWS Cost Explorer
Cost Explorer provides visualizations of your AWS spending by service, account, region, tag, and time period. Enable it immediately on every account. Configure monthly budget alerts. Review weekly. It's free for standard usage.
Cost ExplorerAWS BudgetsCost Anomaly Detection
Practice 02
Implement Cost Allocation Tags
Without tags, you cannot tell which team, project, or environment is spending what. Define mandatory tags (Project, Environment, Owner, CostCenter) and enforce them via SCPs. Cost Explorer uses tags to split costs by any dimension you define.
Cost Allocation TagsTag PoliciesAWS Config
Practice 03
Set Budgets & Alerts
Create AWS Budgets for each account, team, and project. Set alerts at 50%, 80%, and 100% of budget. Configure cost anomaly detection to alert on unexpected spikes (e.g., someone accidentally provisions 100 GPU instances). Never discover overspend in the monthly invoice.
AWS BudgetsCost Anomaly DetectionSNS Alerts
Practice 04
Use AWS Cost & Usage Report (CUR)
CUR is the most granular cost data available — line-item level detail for every resource, every hour, every day. Store in S3, query with Athena, visualize with QuickSight or Grafana. Essential for building custom cost dashboards for clients and finance teams.
CURAthenaQuickSight
Cost Optimization · Compute
Rightsizing Best Practices
Rightsizing means matching resource size to actual workload requirements. The average AWS customer runs compute at 10–15% average CPU utilization — meaning 85–90% of compute spend is waste.
Practice 01
Use AWS Compute Optimizer
Compute Optimizer analyzes CloudWatch metrics for EC2, EBS, Lambda, and ECS and recommends the optimal instance type and size. It can identify instances that are over-provisioned by 2–4x. Typical saving: 25–40% on EC2 costs alone.
Compute OptimizerCloudWatch Metrics
Practice 02
Use Graviton (ARM) Instances
AWS Graviton3 processors deliver 40% better price/performance than comparable x86 instances. Supported by most managed services (RDS, ElastiCache, Lambda, ECS). If your application supports Linux ARM64, switching to Graviton is an immediate 20–40% cost reduction.
Graviton3t4g/m7g/c7gARM64
Practice 03
Lambda Over EC2 for Event-Driven Work
Lambda charges only for execution time (per millisecond). An EC2 instance runs 24/7 whether processing or not. For workloads that run <20% of the time, Lambda is almost always cheaper — and removes the management overhead entirely.
AWS LambdaStep FunctionsEventBridge
Practice 04
ECS Fargate for Containers
Fargate eliminates the need to manage EC2 instances for container workloads. Pay per vCPU/memory per second. For bursty or infrequent workloads, Fargate is cheaper than maintaining EC2 capacity. Use Fargate Spot for fault-tolerant batch workloads at 70% discount.
FargateFargate SpotECS

Cost Optimization · Culture
FinOps Culture Best Practices
FinOps is not a tooling problem — it is a cultural and organizational practice. Technology alone cannot fix cloud cost problems without people, process, and accountability structures.
1
Inform Phase — Visibility & Allocation
Every team must be able to see their cloud costs in near-real-time. Implement cost allocation tags, chargeback or showback models, and unit economics (cost per customer, cost per transaction). When teams see their spend, behavior changes immediately.
2
Optimize Phase — Continuous Improvement
Run monthly cost reviews with each engineering team. Use Savings Plans and Reserved Instances for predictable workloads. Challenge every "always-on" resource. Build a waste elimination backlog and track it like technical debt.
3
Operate Phase — Real-Time Decision Making
Embed cost awareness into engineering workflows. Define cost budgets per sprint. Create anomaly alerts that trigger same-day investigation. Make cost a first-class engineering metric alongside performance and reliability.

Multi-Account Strategy Best Practices
Using a single AWS account for everything is the most common governance mistake. Multi-account architecture is the foundation of every enterprise cloud governance model.
✅ Why Multi-Account? Security blast radius containment (a breach in one account cannot spread to others), separate billing per team/project, independent service limits, clean environment separation (dev/staging/prod), and regulatory compliance isolation.

Governance · Policy
Cloud Policy & Standards Best Practices
Governance without documented policy is opinion. Policy without enforcement is decoration. Effective cloud governance requires written standards AND automated enforcement.
Practice 01
Service Control Policies (SCPs)
SCPs set the maximum permissions boundary for all accounts in your organization. Use them to: deny use of unapproved regions, prevent disabling security services (GuardDuty, CloudTrail), block creation of public resources, enforce encryption requirements.
AWS OrganizationsSCPsPreventive Controls
Practice 02
AWS Config Rules for Compliance
AWS Config continuously evaluates your resource configurations against defined rules. Use managed rules (200+ available) and custom rules. Automatically remediate non-compliant resources using SSM Automation. Build compliance dashboards for executive reporting.
AWS ConfigConfig RulesAuto Remediation
Practice 03
AWS Control Tower Guardrails
Control Tower provides pre-built governance guardrails categorized as Mandatory (always enforced), Strongly Recommended, and Elective. Preventive guardrails use SCPs. Detective guardrails use Config rules. Start with all Mandatory + Strongly Recommended.
Control TowerGuardrailsLanding Zone
Practice 04
Service Catalog for Approved Architectures
AWS Service Catalog lets you define approved CloudFormation templates (products) that teams can self-service deploy. This gives teams speed and autonomy while ensuring they only deploy architectures that meet security, cost, and compliance standards.
Service CatalogCloudFormationSelf-Service

Governance · Operations
Cloud Operations Best Practices
Operational excellence means running cloud workloads reliably and efficiently, with the ability to detect problems early, respond quickly, and learn from failures.
Practice 01
Observability — Metrics, Logs, Traces
Every production system needs: metrics (CloudWatch), structured logs (CloudWatch Logs / OpenSearch), and distributed tracing (AWS X-Ray). Without all three, diagnosing production incidents is guesswork. Define dashboards for the 4 Golden Signals: latency, traffic, errors, saturation.
CloudWatchX-RayOpenSearch
Practice 02
Infrastructure as Code — Always
Every resource in production must be defined in code. Manual console changes are immediately a drift risk. Enforce this with AWS Config drift detection. Use CloudFormation StackSets to deploy consistent configurations across all accounts and regions simultaneously.
CloudFormationCDKStackSets
Practice 03
Patch Management at Scale
Use AWS Systems Manager Patch Manager to automate OS patching across all EC2 instances. Define patch baselines per OS and environment. Schedule maintenance windows. Never rely on humans to manually patch production servers — it never happens consistently.
SSM Patch ManagerMaintenance WindowsPatch Baselines
Practice 04
Runbooks & Playbooks
Document every operational procedure: deployment runbooks, incident response playbooks, DR activation procedures, access request processes. Use AWS Systems Manager OpsCenter to link runbooks directly to operational issues. On-call engineers shouldn't be guessing at 2 AM.
SSM OpsCenterSSM DocumentsAutomation
