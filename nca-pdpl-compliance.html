NCA CCC · Access Control Domain
Identity & Access Management Controls
NCA Cybersecurity Controls require strict access management aligned with least privilege. All controls must be documented and demonstrable during NCA audits.
🔒 NCA CCC Reference: These controls map to NCA CCC Domain 3 — Access Control. Failure to implement these can result in non-compliance findings during official NCA assessments.
🔐 IAM Controls
0 / 5 complete
  • NCA-IAM-01
    Root Account MFA & Restricted Usage
    AWS root account must have hardware or virtual MFA enabled. Root credentials must be stored in a secure vault. Root account must not be used for day-to-day operations. Documented root usage policy must exist.
    AWS RootMFAIAM Policy
  • NCA-IAM-02
    Least Privilege Access Policy
    All IAM roles and users must follow least privilege. No wildcard (*) actions in production policies. IAM Access Analyzer must be enabled to identify overly permissive policies. Quarterly access reviews must be conducted and documented.
    IAM PoliciesAccess AnalyzerQuarterly Review
  • NCA-IAM-03
    MFA Enforced for All Human Users
    MFA must be enforced via IAM policy (deny all actions if MFA not present) for all human IAM users. AWS IAM Identity Center must require MFA. Hardware MFA tokens required for privileged admin accounts. Document MFA enforcement policy.
    MFA PolicyIAM Identity CenterPrivileged Access
  • NCA-IAM-04
    No Long-Term Access Keys
    Access keys must not be used for EC2, Lambda, or ECS workloads — IAM roles must be used instead. Any remaining access keys must be rotated every 90 days maximum. AWS Config rule iam-access-keys-rotated must be active and reporting green.
    Access KeysIAM RolesAWS Config
  • NCA-IAM-05
    Service Control Policies (SCPs) Enforced
    AWS Organizations SCPs must enforce: deny use of non-approved regions, deny disabling CloudTrail or GuardDuty, deny creation of public S3 buckets, deny IAM changes without MFA. SCPs must be documented and reviewed semi-annually.
    AWS OrganizationsSCPsPreventive Controls
NCA CCC · Data Security Domain
Data Security Controls
NCA requires protection of data at rest and in transit with documented encryption standards and key management procedures.
🗄️ Data Security Controls
0 / 5 complete
  • NCA-DAT-01
    Encryption at Rest — All Storage Services
    S3 buckets: SSE-KMS enabled by default. EBS volumes: encrypted using KMS CMK. RDS instances: encryption enabled at creation. ElastiCache: encryption at rest enabled. AWS Config rule: encrypted-volumes must report compliant.
    KMSS3 SSEEBS EncryptionRDS Encryption
  • NCA-DAT-02
    Encryption in Transit — TLS 1.2+ Enforced
    All ALBs: HTTPS listeners only, HTTP redirects to HTTPS. API Gateway: TLS 1.2 minimum policy. S3 bucket policies: deny non-HTTPS access (aws:SecureTransport condition). RDS: require_ssl parameter enabled. Document TLS policy version standards.
    TLS 1.2+ACMHTTPS Enforcement
  • NCA-DAT-03
    AWS KMS Key Management Policy
    Customer Managed Keys (CMKs) must be used for sensitive data. Key rotation must be enabled (annual automatic rotation). Key policies must follow least privilege. Key usage must be auditable via CloudTrail. Key administrators and key users must be separate roles.
    KMS CMKKey RotationKey Policy
  • NCA-DAT-04
    S3 Block Public Access — Account Level
    Block Public Access must be enabled at the account level AND each bucket level. AWS Config rule s3-bucket-public-read-prohibited and s3-bucket-public-write-prohibited must be active. Results must show 100% compliant. Any exceptions require documented business justification and CISO approval.
    S3 BPAAWS ConfigAccount Level
  • NCA-DAT-05
    Secrets & Credentials in AWS Secrets Manager
    All database passwords, API keys, certificates, and service credentials must be stored in AWS Secrets Manager — never in code, environment variables, or SSM Parameter Store (unencrypted). Automatic rotation must be enabled where supported. Secret access must be audited via CloudTrail.
    Secrets ManagerRotationNo Plaintext Secrets
NCA CCC · Network Security Domain
Network Security Controls
Network segmentation and perimeter controls are mandatory NCA requirements for all cloud-hosted systems processing government or sensitive data.
🌐 Network Controls
0 / 4 complete
  • NCA-NET-01
    VPC Network Segmentation (3-Tier)
    All workloads must be deployed in a VPC with clear network segmentation: Public tier (load balancers only), Private tier (application servers), Data tier (databases, caches). No database or application instances should have public IP addresses. Network diagrams must be documented and current.
    VPCSubnets3-Tier Architecture
  • NCA-NET-02
    AWS WAF Deployed on Public Endpoints
    AWS WAF must be deployed in front of all internet-facing ALBs, CloudFront distributions, and API Gateways. AWS Managed Rule Groups (Core Rule Set, Known Bad Inputs) must be enabled. Rate limiting rules must be configured. WAF logs must be shipped to S3 and retained for minimum 1 year.
    WAFManaged RulesRate Limiting
  • NCA-NET-03
    VPC Flow Logs Enabled
    VPC Flow Logs must be enabled for all VPCs, capturing ACCEPT and REJECT traffic. Logs must be delivered to CloudWatch Logs and/or S3. Retention: minimum 90 days hot storage, 1 year archive. Flow logs must be monitored for anomalies and unauthorized access attempts.
    VPC Flow LogsCloudWatch1-Year Retention
  • NCA-NET-04
    No Direct SSH/RDP Access to Production
    Port 22 (SSH) and 3389 (RDP) must not be open to 0.0.0.0/0 on any production Security Group. All administrative access must use AWS Systems Manager Session Manager (no open ports required, full audit trail). This must be verified via AWS Config rule restricted-ssh.
    Session ManagerSecurity GroupsAWS Config
NCA CCC · Monitoring Domain
Monitoring & Audit Controls
Comprehensive audit logging and continuous monitoring are mandatory. NCA auditors will request evidence of log retention, monitoring coverage, and alert response times.
📊 Monitoring & Audit Controls
0 / 5 complete
  • NCA-MON-01
    AWS CloudTrail Multi-Region Trail
    A multi-region CloudTrail trail must be active and logging all management and data events across all AWS accounts. Log file validation must be enabled to detect tampering. Logs delivered to a dedicated S3 bucket with MFA Delete enabled. CloudTrail must not be modifiable by non-security-admin roles.
    CloudTrailMulti-RegionLog Integrity
  • NCA-MON-02
    AWS GuardDuty Enabled All Accounts
    GuardDuty must be enabled in every AWS account and every region. GuardDuty findings must be centralized to a Security account using GuardDuty delegated administrator. All HIGH severity findings must trigger automated notification within 15 minutes. Finding closure must be documented.
    GuardDutyAll AccountsAll Regions
  • NCA-MON-03
    AWS Security Hub Enabled & Configured
    Security Hub must be enabled with CIS AWS Foundations Benchmark standard active. Score must be above 80% compliance. Findings must be reviewed weekly minimum. Critical findings must be remediated within 24 hours. Security Hub findings must feed into an operational dashboard reviewed by security team daily.
    Security HubCIS Benchmark80%+ Score
  • NCA-MON-04
    AWS Config Rules — Continuous Compliance
    AWS Config must be enabled in all accounts and regions. Minimum required rules: encrypted-volumes, restricted-ssh, s3-bucket-public-read-prohibited, iam-root-access-key-check, cloudtrail-enabled, mfa-enabled-for-iam-console-access. Non-compliant resources must trigger auto-remediation or immediate notification.
    AWS ConfigCompliance RulesAuto Remediation
  • NCA-MON-05
    Log Retention Policy — Minimum 1 Year
    All security-relevant logs (CloudTrail, VPC Flow, ALB Access, WAF, GuardDuty) must be retained for minimum 1 year: 90 days in CloudWatch Logs (hot), remaining in S3 (warm/cold). S3 lifecycle policies must automate tier transitions. Log deletion must require MFA and security-admin approval.
    1-Year RetentionS3 LifecycleCloudWatch Logs
NCA CCC · Incident Response Domain
Incident Response Controls
NCA requires a documented, tested incident response capability. Evidence of IR plan existence, testing frequency, and response times are reviewed during assessments.
🚨 Incident Response Controls
0 / 3 complete
  • NCA-INC-01
    Documented Cloud Incident Response Plan
    A written Cloud Incident Response Plan (IRP) must exist and cover: classification levels (P1–P4), escalation paths, notification requirements (internal + NCA reporting), containment procedures, evidence preservation, and post-incident review process. Plan must be approved by CISO and reviewed annually.
    IRP DocumentClassificationCISO Approved
  • NCA-INC-02
    Automated Incident Detection & Alerting
    GuardDuty HIGH findings must trigger automated SNS notifications to the security team within 15 minutes. EventBridge rules must auto-isolate compromised EC2 instances. A 24/7 on-call security rotation must be documented. Mean Time to Detect (MTTD) must be tracked and reported monthly.
    GuardDutyEventBridge24/7 On-Call
  • NCA-INC-03
    Annual Incident Response Testing (Tabletop)
    A full tabletop exercise or simulation of a security incident must be conducted annually. Scenarios must include: data breach, ransomware, account compromise, and insider threat. Results must be documented with lessons learned and improvement action items tracked to closure. Evidence must be available for NCA audit.
    Tabletop ExerciseAnnualDocumentation
PDPL · Data Residency
Data Residency Controls
PDPL requires that personal data of Saudi residents be stored and processed within Saudi Arabia or approved jurisdictions. Non-compliance carries significant financial penalties.
🇸🇦 Primary Region: AWS Middle East (Bahrain) — me-south-1 is the primary compliant region for Saudi personal data. AWS UAE (ae-central-1) may be acceptable as secondary — confirm with SDAIA guidance and legal counsel for your specific data classification.
📍 Data Residency Controls
0 / 4 complete
  • PDPL-RES-01
    Personal Data Stored in Approved AWS Region
    All personal data as defined by PDPL (name, ID number, contact info, financial data, health data, location data) must be stored in AWS me-south-1 (Bahrain) or another SDAIA-approved region. Data mapping document must identify all services containing personal data and their regions. Verified by configuration review.
    me-south-1Data MappingSDAIA Approved
  • PDPL-RES-02
    SCP Preventing Data Replication to Non-Approved Regions
    AWS Organizations SCP must deny S3 replication, RDS snapshots, and DynamoDB global tables to non-approved regions. S3 bucket replication rules must be audited quarterly. AWS Config rule s3-bucket-replication-enabled must flag any unexpected replication configurations for immediate review.
    SCPsRegion RestrictionS3 Replication Control
  • PDPL-RES-03
    Personal Data Inventory & Classification
    A complete inventory of all personal data assets must be maintained: what data, which systems, what classification level, retention period, legal basis for processing. Amazon Macie must be enabled on all S3 buckets containing personal data to automatically discover and classify sensitive data. Inventory reviewed quarterly.
    Amazon MacieData InventoryClassification
  • PDPL-RES-04
    Data Retention & Deletion Policy Enforced
    Documented retention periods must exist for all personal data categories. S3 Object Lock and lifecycle policies must enforce automated deletion at retention expiry. RDS automated backup retention must not exceed the defined period. Annual data purge must be documented and evidenced. Right to erasure requests must be processable within 30 days.
    S3 Object LockLifecycle PoliciesRight to Erasure
PDPL · Consent & Rights
Consent & Data Subject Rights Controls
PDPL gives Saudi data subjects specific rights over their personal data. Technical controls must enable these rights to be fulfilled within defined timeframes.
🤝 Consent & Rights Controls
0 / 4 complete
  • PDPL-CON-01
    Consent Management System Implemented
    A technical system for capturing, storing, and proving user consent must be implemented. Consent must be explicit, specific, informed, and withdrawable. Consent records must be stored durably (S3 with versioning) with timestamps. Withdrawal of consent must trigger automated downstream data deletion workflows.
    Consent RecordsS3 VersioningWithdrawal Workflow
  • PDPL-CON-02
    Data Subject Access Request (DSAR) Process
    A documented and tested process for handling Data Subject Access Requests must exist. DSARs must be fulfilled within 30 days. Technical capability to export all personal data for a specific individual must be implemented and tested. Process must be documented in the Privacy Notice.
    DSAR Process30-Day SLAData Export
  • PDPL-CON-03
    Privacy Notice Published & Current
    A PDPL-compliant Privacy Notice must be published and accessible. It must state: what data is collected, legal basis, purpose, retention period, third-party sharing, data subject rights, and contact for requests. Notice must be reviewed annually and updated within 30 days of material changes to data processing.
    Privacy NoticeLegal BasisAnnual Review
  • PDPL-CON-04
    Data Breach Notification Process
    A documented data breach notification process must exist. SDAIA must be notified within 72 hours of discovering a personal data breach. Affected data subjects must be notified without undue delay when breach is high-risk. GuardDuty + Macie findings must feed into breach detection workflows with automated escalation.
    72-Hour NotificationSDAIABreach Detection
PDPL · Data Protection
Technical Data Protection Controls
PDPL requires appropriate technical and organizational measures to protect personal data — these are the specific AWS technical controls that demonstrate compliance.
🔒 Data Protection Controls
0 / 5 complete
  • PDPL-PRO-01
    Amazon Macie for Personal Data Discovery
    Amazon Macie must be enabled on all S3 buckets that may contain personal data. Macie findings must be reviewed weekly. Any new S3 buckets must be automatically enrolled in Macie scanning via automation. Macie findings that identify unprotected personal data must trigger immediate remediation workflow.
    Amazon MacieS3 ScanningAuto Enrollment
  • PDPL-PRO-02
    Pseudonymization / Anonymization for Non-Production
    Personal data must never be copied to dev or test environments without anonymization or pseudonymization. A documented data masking process must exist. Production database dumps must be scrubbed before use in lower environments. This must be enforced technically — not just by policy.
    Data MaskingAnonymizationNon-Prod Control
  • PDPL-PRO-03
    Data Access Logging for Personal Data Stores
    All access to systems containing personal data must be logged. RDS: enable Enhanced Monitoring + audit log. DynamoDB: enable CloudTrail data events. S3: enable server access logging + CloudTrail data events. Logs must be retained 1 year and reviewed for anomalous access patterns.
    RDS Audit LogsCloudTrail Data EventsS3 Access Logs
  • PDPL-PRO-04
    Privacy Impact Assessment (PIA) for New Systems
    A Privacy Impact Assessment must be completed before deploying any new system that processes personal data. PIA must assess: data flows, risks, controls, legal basis. PIAs must be documented, approved by DPO/Legal, and stored. AWS Well-Architected security review must be linked to PIA for cloud workloads.
    PIA ProcessDPO ApprovalWAF Security Review
  • PDPL-PRO-05
    Third-Party Processor Agreements (DPAs)
    Data Processing Agreements must be in place with all third-party processors of personal data, including AWS (via AWS DPA), SaaS providers, analytics tools, and support vendors. DPAs must be reviewed annually. A register of all processors must be maintained. AWS Artifact provides AWS compliance documents on demand.
    AWS DPAAWS ArtifactProcessor Register
PDPL · Cross-Border Transfer
Cross-Border Data Transfer Controls
PDPL restricts transfer of personal data outside Saudi Arabia. Where cross-border transfers are necessary, specific conditions and safeguards must be in place.
🌍 Cross-Border Transfer Controls
0 / 3 complete
  • PDPL-TRN-01
    Cross-Border Transfer Register & Legal Basis
    All cross-border personal data transfers must be documented in a transfer register: destination country, data categories, legal basis, safeguards in place. Legal basis options: SDAIA adequacy decision, appropriate safeguards (Standard Contractual Clauses equivalent), or explicit consent. Register reviewed quarterly by DPO.
    Transfer RegisterLegal BasisSCCs
  • PDPL-TRN-02
    VPC Endpoints to Prevent Unintended Data Transfer
    VPC Endpoints (Gateway and Interface) must be used for all AWS service access to ensure data does not traverse the public internet and stays within the AWS backbone. S3 bucket policies must include condition: aws:sourceVpce to restrict access to approved VPC endpoints only, preventing unauthorized data movement.
    VPC EndpointsPrivateLinkBucket Policy Conditions
  • PDPL-TRN-03
    AWS CloudFront Origin Shield in Approved Region
    If CloudFront is used to serve content that includes personal data, Origin Shield must be configured to use a location in or nearest to Saudi Arabia. Cache behaviors must be reviewed to ensure personal data is not cached at unapproved edge locations. Signed URLs/cookies must be used for private personal content.
    CloudFrontOrigin ShieldSigned URLs

0 / 4 complete
  • NDMO-CLS-01
    Data Classification Tags on All Resources
    All AWS resources containing data must have a DataClassification tag (Public / Internal / Confidential / Restricted). SCPs must enforce different controls per classification level. AWS Config must flag resources without classification tags. Tag policies must enforce valid values only.
    Classification TagsTag PoliciesSCPs
  • NDMO-CLS-02
    Confidential Data — CMK Encryption Required
    All data classified as Confidential or Restricted must be encrypted with Customer Managed Keys (CMK) — not AWS managed keys. Key policies must restrict key usage to approved roles. Key usage must be logged and auditable. Annual key review must be conducted.
    CMKKMS Key PolicyAnnual Review
  • NDMO-CLS-03
    Restricted Data — Dedicated Tenancy Assessment
    Any data classified as Restricted must be assessed for whether dedicated tenancy (Dedicated Hosts or Dedicated Instances) is required. Assessment must be documented with CISO sign-off. If dedicated tenancy is required, it must be enforced via IAM policy condition ec2:Tenancy.
    Dedicated HostsTenancy PolicyCISO Sign-off
  • NDMO-CLS-04
    Annual Data Classification Review
    All data classification assignments must be reviewed annually for accuracy. New data assets must be classified within 30 days of creation. Classification changes must go through a formal change management process. Classification register must be maintained and accessible to auditors.
    Annual Review30-Day WindowChange Management

NDMO · Data Governance
Data Governance Controls
NDMO requires formal data governance structures to manage government data across its lifecycle in cloud environments.
🏛️ Data Governance Controls
0 / 4 complete
  • NDMO-GOV-01
    Data Steward Assignments
    Named data stewards must be assigned for all critical data domains in the cloud environment. Steward responsibilities must be documented: classification decisions, quality oversight, access approvals, retention decisions. Steward assignments must be tracked and updated on personnel changes.
    Data StewardsResponsibilitiesDocumented
  • NDMO-GOV-02
    AWS Lake Formation for Data Catalog
    AWS Glue Data Catalog or Lake Formation must be implemented to catalog all data assets in the cloud. Catalog must include: data location, schema, classification, owner, sensitivity level. Data lineage must be trackable. Catalog must be searchable and accessible to authorized data stewards.
    AWS GlueLake FormationData Catalog
  • NDMO-GOV-03
    Data Quality Standards & Monitoring
    Data quality standards must be defined for critical data domains: completeness, accuracy, consistency, timeliness, uniqueness. AWS Glue DataBrew or custom Lambda functions must monitor quality metrics. Quality dashboards must be reviewed by data stewards monthly. Quality issues must be tracked to resolution.
    Data QualityGlue DataBrewMonthly Review
  • NDMO-GOV-04
    Data Governance Committee (DGC) Charter
    A formal Data Governance Committee must exist with a documented charter: scope, membership, meeting cadence (minimum quarterly), decision authority, escalation path. DGC must approve: classification policies, sharing agreements, major architecture changes, data retention changes. Minutes must be recorded and retained 3 years.
    DGC CharterQuarterly MeetingsDecision Authority
Cloud Operations · Hardening
Configuration & Hardening Controls
Secure configuration baselines and continuous compliance checks are the foundation of cloud operational security.
⚙️ Config & Hardening Controls
0 / 5 complete
  • OPS-CFG-01
    CIS AWS Foundations Benchmark — 80%+ Score
    AWS Security Hub CIS AWS Foundations Benchmark v1.4 must be enabled and score must exceed 80%. All CRITICAL and HIGH findings must be remediated within defined SLAs (Critical: 24h, High: 7 days). Score must be tracked monthly and trending upward. Score must be reportable to auditors on demand.
    Security HubCIS Benchmark80%+ Score
  • OPS-CFG-02
    AWS Inspector for Vulnerability Scanning
    AWS Inspector must be enabled on all EC2 instances and Lambda functions. Inspector must scan for OS vulnerabilities and software composition vulnerabilities. Critical CVEs must be remediated within 72 hours. High CVEs within 7 days. Inspector findings must integrate with Security Hub for centralized tracking.
    AWS InspectorCVE Scanning72h Critical SLA
  • OPS-CFG-03
    Infrastructure as Code — All Production Resources
    100% of production infrastructure must be defined in CloudFormation, CDK, or Terraform. No manual console changes permitted in production. AWS Config drift detection must alert on any unmanaged resource changes. CloudFormation stack policies must prevent destructive updates without explicit override.
    CloudFormationCDKDrift Detection
  • OPS-CFG-04
    EC2 Image Builder — Hardened AMIs
    All EC2 instances must launch from hardened AMIs created by EC2 Image Builder. AMI pipeline must apply CIS benchmark hardening, install required agents (SSM, CloudWatch), and disable unnecessary services. AMIs must be rebuilt monthly to include latest patches. Old AMIs must be deregistered after replacement.
    EC2 Image BuilderHardened AMIsMonthly Rebuild

    AWS Config Conformance Packs Deployed
    AWS Config Conformance Packs must be deployed across all accounts. Minimum required: Operational Best Practices for NCA, Operational Best Practices for Security. Custom conformance pack for organization-specific controls must be maintained. Conformance pack compliance reports must be reviewed monthly.
    Conformance PacksAll AccountsMonthly Review
Cloud Operations · Patching
Patch & Vulnerability Management
Unpatched systems are the #2 cause of cloud security breaches after IAM misconfiguration. Automated, enforced patching is non-negotiable.
🔧 Patch & Vuln Controls
0 / 4 complete
  • OPS-PAT-01
    SSM Patch Manager — Automated Patching
    AWS Systems Manager Patch Manager must be configured for all EC2 instances. Patch baselines must be defined per OS with Critical patches auto-approved after 3 days. Maintenance windows must be defined per environment (prod: weekend, non-prod: weeknight). Patch compliance reports must be reviewed weekly.
    SSM Patch ManagerPatch BaselinesMaintenance Windows
  • OPS-PAT-02
    Patch Compliance Targets — 95%+ within 30 Days
    Patch compliance target: 100% Critical patches within 7 days, 95% Important patches within 30 days. Compliance must be measured via SSM Patch Manager dashboards. Non-compliant instances must be escalated after 7 days. Exceptions must be formally approved with compensating controls documented.
    95% Target7-Day CriticalException Process
  • OPS-PAT-03
    Container Image Scanning in CI/CD Pipeline
    Amazon ECR image scanning must be enabled (enhanced scanning with Inspector). CI/CD pipelines must fail builds that include container images with Critical CVEs. Images must be rescanned weekly even without code changes. Base images must be rebuilt monthly from official sources.
    ECR ScanningInspectorCI/CD Gate
  • OPS-PAT-04
    RDS Auto Minor Version Upgrade Enabled
    RDS instances must have auto minor version upgrade enabled to receive security patches automatically during maintenance windows. Major version upgrade plan must be maintained for all RDS instances — no instance should run on an end-of-life engine version. Upgrade plan reviewed semi-annually.
    RDS Auto UpgradeMaintenance WindowsEOL Planning
Cloud Operations · Backup
Backup & Recovery Controls
Documented, automated, and tested backup procedures are required by NCA, PDPL, and NDMO for all systems processing government or personal data.
💾 Backup & Recovery Controls
0 / 4 complete
  • OPS-BAK-01
    AWS Backup — Centralized Backup Policy
    AWS Backup must be configured with backup plans covering all critical resources: RDS, EC2, EFS, DynamoDB, S3. Backup frequency must match RPO targets. Backup retention must meet regulatory minimums (NCA: 1 year for audit logs, PDPL: aligned with data retention policy). Cross-region backup copies must exist for critical systems.
    AWS BackupCross-Region CopyRPO Aligned
  • OPS-BAK-02
    Backup Encryption with CMK
    All backups must be encrypted using Customer Managed KMS keys — not default AWS Backup keys. Backup vault access policies must restrict who can delete backups. Vault Lock must be enabled for audit/compliance backups to make them immutable. Cross-account backup copies add additional protection against ransomware.
    Backup EncryptionVault LockCross-Account
  • OPS-BAK-03
    Quarterly Backup Restoration Testing
    Backup restoration must be tested quarterly for all critical systems. Tests must verify: restoration completes within RTO target, restored data is consistent and complete, restored system is functional. Test results must be documented with date, system, RTO achieved, and any issues. Evidence required for NCA/NDMO audit.
    Quarterly TestingRTO VerificationDocumented Evidence
  • OPS-BAK-04
    RTO & RPO Targets Documented & Met
    Formal RTO and RPO targets must be defined, approved, and documented for all production systems. Targets must be tested annually at minimum via DR exercise. Actual recovery times from any incidents or tests must be logged. Systems failing to meet targets must have improvement plans. RTO/RPO must be reviewed when systems change.
    RTO/RPODR ExerciseAnnual Testing
